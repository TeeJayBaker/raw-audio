# TL;DR — why this matters for raw-audio CPU gen

pMF is **the first 1-NFE pixel-space diffusion/flow result that's actually competitive with latent-space SOTA**: 2.22 FID on ImageNet 256² and 2.48 FID on 512², single forward pass, no VAE, no teacher, no GAN. It does this by stacking three pieces we already individually believe in: (1) MeanFlow's average-velocity training (compute `u` between two time points via the MeanFlow identity, a JVP-with-stop-grad-on-`du/dt`); (2) JiT-style **x-prediction** — the network directly outputs an "image-like" quantity that's hypothesised to lie on a low-d manifold, then a conversion `x → u → v` puts the loss back in velocity-space (FM v-loss); and (3) **perceptual loss** (LPIPS / ConvNeXt-V2) directly on `x_θ`, which is only possible because the network output is in pixel space. The recipe is small, principled, and most importantly **everything in it is dimension-agnostic** — it ports to raw 1D audio almost verbatim once we swap DiT for a 1D backbone and LPIPS for an audio perceptual loss. This is the strongest known evidence so far that one-step pixel-space generation can be done in a fully end-to-end network without external tokenizers — directly relevant to "drop the VAE/codec floor and generate waveform end-to-end" as a research bet.

# What problem does it solve

Modern diffusion/flow generators are characterised by two cost drivers: **multi-step sampling** and **latent-space operation**. Recent work has separately attacked each:

- Multi-step → one-step: consistency models, MeanFlow / iMF / Drifting (from-scratch 1-NFE).
- Latent-space → pixel-space: JiT, PixelFlow, DiP, PixelDiT, EPG.

But the joint problem — "1-NFE *and* pixel-space" — is much harder, because a single network now has to do all of (compress, denoise, generate). The only prior credible 1-NFE pixel-space result (excluding GANs) was EPG-L/16 at **FID 8.82** on ImageNet-256, which still needed an expensive SSL pre-training stage. pMF closes that gap dramatically: **FID 2.22 at 1-NFE pixel-space, single end-to-end training stage, no SSL pretrain**, beating prior 1-NFE pixel-GAN entries (StyleGAN-XL 2.30, GigaGAN 3.45) at a fraction of the GFLOPs.

# Method, the part that distinguishes it from neighbours

## The core observation: separate "output space" from "loss space"

The paper's central guideline: **the network output target and the regression loss don't have to live in the same space**. Standard practice glues them together (DiT predicts ε with ε-loss; SiT predicts v with v-loss; MeanFlow predicts u with u-loss; JiT predicts x with x-loss). pMF deliberately decouples them:

- **Prediction space:** `x` — a "denoised image" quantity that's hypothesised to lie on the data manifold and therefore be easier for a finite-capacity network to model.
- **Loss space:** `v` — instantaneous velocity, via a conversion `x → u → v` and the MeanFlow v-loss formulation.

Table 1 from the paper compares the layout across methods:

| Method | Prediction space | Conversion | Loss space |
|---|---|---|---|
| DiT | ε | — | ε |
| SiT | v | — | v |
| MeanFlow (MF) | u | — | u |
| iMF | u | u → v | v |
| JiT | x | x → v | v |
| **pMF** | **x** | **x → u → v** | **v** |

pMF takes JiT's "predict image, loss in velocity" and substitutes JiT's loss-space (`v`) for iMF's MeanFlow `v`-loss (which uses average velocity `u` plus a stop-grad JVP). The conversion chain is one extra arithmetic step but no extra network calls.

## The denoised image field x(z_t, r, t)

In standard flow-matching, the linear interpolation is `z_t = (1 − t) x + t ε` (with `t = 0` at data, `t = 1` at noise; note the convention is flipped vs JiT). Average velocity is

```
u(z_t, r, t) = (1/(t − r)) ∫_r^t v(z_τ, τ) dτ
```

for `0 ≤ r ≤ t ≤ 1`. pMF defines a **2D field** indexed by `(r, t)`:

```
x(z_t, r, t) ≜ z_t − t · u(z_t, r, t)        (Eq. 8)
```

The two boundary cases recover known objects:

- `r = t` (degenerate): `u = v`, so `x(z_t, t, t) = z_t − t · v(z_t, t)` — exactly JiT's x-prediction target. Interpretable as "the denoised image given the *instantaneous* velocity at z_t".
- `r = 0` (full integration): `x(z_t, 0, t) = z_0` — the actual endpoint of the ODE trajectory, drawn from `p_data`.

Empirically (Fig. 1, right): as `r → 0` the field `x(z_t, r, t)` looks like a **clean or mildly-blurred image**, regardless of `t`. This is the **generalised manifold hypothesis**: for any (r, t), `x` approximately lives on the data manifold, while `u` and `v` are necessarily noisy (they have to carry the noise component of `z_t`). Modelling `x` is therefore easier than modelling `u` or `v` for a network with finite capacity.

## Training algorithm

Pseudocode (Alg. 1, lightly annotated):

```python
# net: x-prediction network
# x: training batch in pixels
t, r = sample_t_r()                            # (r, t) in 0 <= r <= t <= 1
e    = randn_like(x)
z    = (1 - t) * x + t * e                     # FM linear interpolant

def u_fn(z, r, t):
    return (z - net(z, r, t)) / t              # u from x-pred (Eq. 11)

v = u_fn(z, t, t)                              # instantaneous velocity at (t, t)
u, dudt = jvp(u_fn, (z, r, t), (v, 0, 1))      # MeanFlow JVP, stop-grad on dudt
V = u + (t - r) * stopgrad(dudt)               # iMF compound function V_θ (Eq. 7)
loss = metric(V, e - x)                        # FM v-loss against ground-truth (e - x)
```

So at heart: x-prediction network, MeanFlow JVP for the velocity-side compound function, standard FM v-loss `||V_θ − (ε − x)||²` against the target `ε − x`. Network capacity is spent learning the "easy" manifold-bound quantity `x`; the loss is computed in velocity space where it's well-conditioned.

CFG follows iMF: the CFG scale `ω` and interval `[ω_lo, ω_hi]` are conditioning inputs to the network, so inference is still **1 NFE even with CFG** (no doubled forward pass).

## Perceptual loss directly on x_θ

Because `x_θ` is itself a pixel-space image, the **perceptual loss family (LPIPS, ConvNeXt-V2-based) applies directly**:

```
L = L_pMF + λ · L_perc(x_θ, x_clean)
```

Applied only when `t ≤ t_thr = 0.8` so the predicted denoised image isn't too blurry to feed to a perceptual network. They use both VGG-LPIPS (λ=0.4) and a ConvNeXt-V2 variant (λ=0.1); combining them gives the largest jump (Fig. 3b: 9.56 → 5.62 → **3.53** FID).

This is a benefit pMF gets *for free* by being in pixel space — latent-space methods can only apply perceptual loss during VAE training, not at generator training. Direct analogue for audio: any waveform-domain perceptual loss (multi-resolution STFT loss, AudioCLIP-style embedding distance) plugs straight in here, only possible because `x_θ` is raw waveform.

## Why x-prediction (not u-prediction or v-prediction)

Ablation (Tab. 2):
- 64×64 (patch dim 48): u-pred 3.82, x-pred 3.80 → indistinguishable at low input dimensionality.
- **256×256 (patch dim 768): u-pred 164.89, x-pred 9.56** → catastrophic collapse for u-prediction; x-prediction works.

The toy experiment (Fig. 2) reinforces this: project a 2D Swiss roll into a D-dim observation space (D ∈ {2, 8, 16, 512}); u-prediction degrades smoothly with D, x-prediction is robust through D = 512. The manifold hypothesis pays off precisely when the observation space is high-dimensional relative to the true data manifold.

This is **the central message for our project**: raw audio is exactly such a regime. A 1 s 16 kHz mono clip is 16 000-dim, but the music manifold is plausibly <100-dim. We should predict the (low-d) sample-space object, not the (high-d) velocity field.

## What did *not* work (Tab. 3a)

Pre-conditioners — Karras-style `x_θ = c_skip z_t + c_out · net_θ`:
- Linear (`c_skip = 1 − t`): **FID 34.61** (terrible)
- EDM-style: 14.43
- sCM-style: 13.81
- **No pre-conditioning (raw x-pred): 3.53**

Pre-conditioners deviate the network output from the manifold (`c_skip · z_t` adds noise back in), defeating the whole point. In *low*-d settings (where MeanFlow / iMF / EDM live) this didn't matter; in high-d pixel space it matters a lot. Lesson: keep the output target on the manifold — don't blend noise back in.

Time samplers — restricting `(r, t)` to lines:
- Only `r = t` (Flow Matching subspace): 194.53
- Only `r = 0` (CM-like subspace): 389.28
- Both lines together: 106.59
- **Full triangle `0 ≤ r ≤ t`: 3.53**

So the MeanFlow JVP across `(r, t)` is genuinely doing work — the field needs to be regressed across the entire plane, not just on its boundaries.

# Architecture

Standard iMF / DiT-style transformer; nothing exotic:

| Variant | Depth | Hidden dim | Heads | Patch | Seq len | Params | GFLOPs |
|---|---|---|---|---|---|---|---|
| pMF-B/16 (256²) | 16 | 768 | 12 | 16×16 | 16² = 256 | 118 M | 33 |
| pMF-L/16 (256²) | 32 | 1024 | 16 | 16×16 | 256 | 413 M | 117 |
| pMF-H/16 (256²) | 48 | 1280 | 16 | 16×16 | 256 | 959 M | 272 |
| pMF-B/32 (512²) | 16 | 768 | 12 | 32×32 | 256 | 120 M | 34 |
| pMF-H/32 (512²) | 48 | 1280 | 16 | 32×32 | 256 | 962 M | 272 |
| pMF-H/64 (1024²) | 48 | 1280 | 16 | 64×64 | 256 | ~970 M | (low) |

Key trick: **sequence length is fixed at 16² = 256 tokens across all resolutions**; high-resolution generation is achieved by increasing the **patch size**, not the token count. At 1024² the per-patch dim hits 64²×3 = 12288 — extremely high — but it still works because the network only needs to predict a manifold-bound quantity. Per-token FLOPs are fixed; only the patchification embedder and the prediction head grow.

Conditioning is **in-context** (concat class / time / `(r, t)` / CFG tokens into the sequence) rather than adaLN-zero — saves ~⅓ params (inherited from iMF).

Other details (Tab. 8): bottleneck dim 128 (B/L) or 256 (H); LPIPS weight 0.4; ConvNeXt weight 0.1; t_thr = 0.8 (or 0.6 for long training); logit-normal `(t, r)` sampler with `r = t` 50% of the time; **Muon optimiser** (not Adam — Fig. 3a shows Muon gives ~3 FID better at 320 epochs); EMA half-lives in {500, 1000, 2000} Mimgs (best is selected post-hoc); batch 1024; lr 1e-3 constant; 160 or 320 epochs.

# Inference cost

**1 NFE — single transformer forward pass on 256 tokens.** No solver, no decoder, no second network at inference. CFG is folded into training as a conditioning input (`ω` is part of the input), so 1 NFE truly means 1 forward pass.

The compute story is unusually clean across resolutions:
- 256²: 272 GFLOPs (H/16)
- 512²: 272 GFLOPs (H/32) — same!
- 1024²: ~273 GFLOPs (H/64) — basically same!

Because the sequence length is fixed at 256 tokens, the cost of the attention + MLP backbone is identical across resolutions; only patchify / unpatchify scales. This is exactly the cost shape we want for variable-length audio (or rather: fixed token budget across variable clip lengths).

Per-image GFLOPs at 256² vs equivalents (1-NFE, fair compute):
- pMF-B/16: 118 M params, **33 GFLOPs**, FID 3.12 — *cheapest 1-NFE result with sub-3.5 FID*
- pMF-L/16: 410 M, 117 GFLOPs, FID 2.52
- pMF-H/16: 956 M, 271 GFLOPs, FID 2.22
- StyleGAN-XL (1-NFE pixel GAN): 166 M, **1574 GFLOPs**, FID 2.30
- Drifting B/16 (1-NFE pixel, prior best): 134 M, 87 GFLOPs, FID 1.76 (still beats pMF-B/16 in quality at 2.6× cost)

No CPU latency reported in the paper. But pMF-B/16 at 33 GFLOPs / 256 tokens / 118 M params is the kind of cost shape that *might* run faster than real time on a CPU at audio sequence lengths if the architecture ports cleanly. That's the experiment.

# How this translates to raw audio

This paper is unusually well-suited for direct audio translation because every piece is dimension-agnostic.

## What ports cleanly

1. **The (output space, loss space) decoupling.** Network predicts a "low-d manifold" quantity `x` (clean waveform), loss is computed via x → u → v in velocity space. Both are well-defined in any dimensionality.
2. **The MeanFlow JVP and v-loss.** No image-specific structure. The training pseudocode runs unchanged if `x` is a 1D waveform tensor.
3. **The patch-size-as-cost-knob trick.** This is *exactly* the move we want for long audio: hold token count at 256 (or 128 or 512), grow patch size to match clip length × sample rate. 1 s @ 22.05 kHz / 256 tokens → patch 86 samples; 10 s @ 22.05 kHz / 256 tokens → patch 860 samples. Per-token transformer FLOPs stay flat across clip lengths. **This is the most important structural lesson.**
4. **Perceptual loss on x_θ.** Direct analogue: multi-resolution STFT loss (Yamamoto et al.), MERT/AudioMAE embedding distance, or a learned waveform-domain perceptual loss. Apply at low `t` only (when `x_θ` is close to clean), exactly as pMF gates LPIPS at `t ≤ 0.8`.
5. **CFG-as-conditioning.** Inference stays 1 NFE — critical for our CPU budget.
6. **In-context conditioning instead of adaLN-zero.** Saves params; portable to a 1D transformer.
7. **No SSL pre-training stage required.** pMF trains end-to-end from scratch — unlike EPG, which needs the contrastive encoder pre-training. This is a major simplification for audio (we don't yet have a good in-house audio φ).

## What needs adaptation

- **Backbone.** DiT → 1D transformer or 1D conv-transformer hybrid. The in-context conditioning works with any token-sequence backbone; if we use a 1D U-Net we'd fall back to FiLM (per iMF translation rules).
- **Perceptual loss choice.** LPIPS / ConvNeXt-V2 are image-specific. We need to pick an audio perceptual loss with similar properties: differentiable, gated on a "clean enough" prediction, comparable across scales. Multi-resolution STFT is the obvious default. The paper's pattern of **combining two perceptual losses with different feature backbones** (VGG-LPIPS + ConvNeXt-V2) is portable: combine MR-STFT + MERT/AudioMAE embedding distance.
- **Patch-size scaling.** For 1D audio: clip length L samples / 256 tokens = patch size P; patch dim is P × C (mono → C=1, stereo → C=2). At long clips this gets large; pMF shows 12288-dim patches (1024² × 3 / 256²) work fine — so a 1 s @ 44.1 kHz mono clip with 256 tokens = 172-dim patches is *trivial* by comparison.
- **t_thr (perceptual gating).** Audio-side equivalent: only apply MR-STFT when `x_θ` is plausibly clean (low `t`). Same gating mechanism.

## Phase, stereo, sample-rate concerns

- **Phase:** because the prediction target `x` is the raw waveform, phase is modelled implicitly — no spectrogram-based information loss. Same advantage CRASH has, but with 1 NFE instead of 50+.
- **Stereo:** patch dim doubles (P × 2 for stereo). No structural change.
- **Sample rate:** the patch-size trick says "raise sample rate → raise patch size, hold token count" — i.e. the same model architecture trains across 16, 22.05, 44.1 kHz. The high-d patches are easy because they live on the low-d audio manifold. **This is a more elegant scaling story for audio than anything CRASH-style has.**

## The natural first experiment

This paper is the **single most direct port-to-audio recipe in our wiki right now**. Concrete plan:

1. Build a 1D transformer (4–6 layers, ~50–100 M params) with iMF-style in-context conditioning.
2. Implement the pMF x-pred-loss-in-v-space training with the MeanFlow JVP. Pseudocode is 12 lines; identical to Alg. 1 with `x` swapped for a 1D waveform tensor.
3. Train on 1 s waveform crops at 22.05 kHz from a small music corpus (256 tokens, patch 86, ~86-dim patches mono).
4. Add MR-STFT perceptual loss at low `t`.
5. Use Muon optimiser (paper shows Muon > Adam by ~3 FID at 320 epochs — worth the small effort).
6. Benchmark CPU single-forward-pass latency.

# Strengths

- **First 1-NFE pixel-space result that's actually SOTA-competitive** (2.22 vs latent iMF 1.72 — gap nearly closed; the prior pixel SOTA at 1 NFE was 8.82). Headline beat.
- **Clean, minimal, end-to-end recipe.** No SSL pretrain (unlike EPG), no teacher (unlike distillation), no GAN, no separate VAE. Single training run.
- **Patch-size-as-resolution-knob with fixed token budget** is a genuinely new architectural primitive — directly applicable to variable-length audio.
- **Pixel-space-enables-perceptual-loss** is a clean argument that latent methods can't replicate, and the perceptual loss accounts for ~6 FID points of the headline.
- **CFG without doubled NFE.** Inference stays 1 forward pass.
- **Cheaper than competitive GANs** by 5–10× compute at equal or better FID.
- The (prediction space, loss space) decoupling is a transferable conceptual primitive — should inform our audio loss design too.

# Limitations / open questions

- **Drifting (2602.04770) still wins on FID at 1 NFE pixel-space:** Drifting L/16 reports FID 1.61 (vs pMF-H/16 2.22), but with a heavier setup (custom latent-MAE+ConvNeXt-V2 feature stack). For from-scratch end-to-end simplicity, pMF wins; for pure quality at 1 NFE, Drifting still wins. They're at different points on the simplicity–quality curve.
- **No CPU latency anywhere.** Headlines are FID and GFLOPs only. We don't know if pMF-B/16's 33 GFLOPs translates to faster-than-RT on a CPU at audio token counts.
- **Muon dependency.** Adam reaches 11.86 FID where Muon reaches 8.71 at 320 epochs — Muon is doing real work, not just speedup. May matter for our small-corpus audio training (Muon is less battle-tested than Adam).
- **EMA half-life selected post-hoc** from {500, 1000, 2000} Mimgs — typical for 1-NFE methods but adds an inference-time selection step.
- **Perceptual loss is image-specific.** The 6-FID gain from LPIPS + ConvNeXt-V2 is the biggest single contributor to the headline number; if no audio analogue is as effective, we lose a meaningful chunk of the quality story.
- **No ablation on dropping the JVP.** It's not clear how much of pMF's quality is the JVP-driven MeanFlow gradient vs the x-pred + perceptual recipe.
- The paper's "generalised manifold hypothesis" is empirical-only — no theory that `x(z_t, r, t)` for general `0 < r < t` is on-manifold; the figures show it visually but no quantitative manifold-distance metric.
