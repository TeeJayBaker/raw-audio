## TL;DR

Large-scale (624 M / 1.03 B-param) multimodal V2A + T2A generator that runs flow matching directly on the **raw 16 kHz / 44.1 kHz waveform** via 200-sample patches + an `x`-prediction `v`-loss + an audio-specific RMS-normalize-and-×3 amplitude lift, on an MMDiT backbone with CLIP + Synchformer conditioning. First production-quality evidence that the pixel-space-FM recipe (JiT-style `x`-pred + patch tokenization) ports to raw audio at scale — matches latent SOTA (MMAudio, HunyuanVideo-Foley) on VGGSound VT2A and AudioCaps T2A. 50 NFE + CFG 4.5; no CPU benchmark; model class is far above the project's CPU budget, but the **recipe** is the portable artifact: raw-waveform patched FM works.

## Why this paper matters for raw sample-space audio gen

Until WavFlow, every published "raw-waveform diffusion / FM" data point at scale was a *conditional vocoder* (HiFi-GAN, WaveFM, Flow2GAN, PeriodWave-Turbo) or a small drum-only demo (CRASH at ~0.48 s). LinDiff is the only earlier patched-ViT on raw waveform — 4-layer / 256-d / 22 kHz vocoder, MOS-class quality. WavFlow is the first time the JiT/pMF image-domain lesson — *patched transformer + `x`-prediction is sufficient for pixel-space generative modelling* — has been instantiated on raw audio at production scale and competitive against latent baselines. The codec floor is no longer hypothetically removable; an actually-removed-it model exists and matches latent SOTA.

The deployment shape is wrong for this project (1 B params, 50 NFE, 8 s clips, 16 / 44.1 kHz) — but the design choices port directly to the smaller, fewer-step backbones this project targets.

## Method

**Flow matching with `x`-prediction.** Standard linear interpolant `x_t = (1−t) x_0 + t x_1` with `x_0 ∼ 𝒩(0, I)`, target velocity `v* = x_1 − x_0`. Network predicts the clean signal `x̂_1 = f_θ(x_t, t, c)`; velocity recovered as `v_θ = (x̂_1 − x_t) / (1 − t)`. Loss is `v`-loss on the recovered velocity:

```
L = E [ ‖ (x̂_1 − x_t)/(1−t) − (x_1 − x_t)/(1−t) ‖² ]
```

This is the JiT/pMF "decouple prediction space (`x`) from loss space (`v`)" pattern. Ablated against pure `v`-pred + `v`-loss (FD_PaSST 77.2) and `x`-pred + `x`-loss (72.7); chosen combination wins (63.05). The `x`-pred + `v`-loss combo specifically wins on **distribution coverage** (IS 15.58 vs 13.99 for `x`-pred + `x`-loss) — confirming pMF's argument that `x`-pred grounds the manifold while `v`-loss preserves the flow-matching gradient geometry.

**Waveform patchify.** Reshape 1-D waveform `x ∈ ℝ^T` into a `C × D` 2-D token grid (`T = C · D`, zero-pad to make divisible, truncate on unpatchify). `D` = samples per token. At 16 kHz / 8 s (`T = 128 000`), `D = 200` → `C = 640` tokens, 12.5 ms granularity per token; at 44.1 kHz, same `D = 200` → `C = 1 764` tokens, 4.5 ms granularity. The 12.5 ms cell is well below the ~25 ms human auditory resolution threshold (Petrini et al. 2009).

The 1-D `T → (C, D)` reshape is parameter-free, deterministic, and analytically invertible (the **only** entry/exit machinery between transformer and waveform — no learned encoder, no neural vocoder). This is the direct analogue of JiT's 2-D `(H, W) → (h, w, P²·C)` pixel patchify.

**Amplitude lifting.** Audio waveforms typically have RMS < 0.2 with sharp zero-centered distributions — most of the dynamic range is unused; signal is "submerged by noise during training" because the loss has near-zero magnitude on quiet samples. WavFlow's fix is one parameter-free preprocessing step:

```
x_lift = s_a · clamp( r_* / rms(x) · x , −1, 1 )
```

with `r_* = 0.33` (RMS target) and `s_a = 3.0` (post-scale). Output is roughly in `[−3, 3]`, closer to the `𝒩(0, 1)` prior the flow needs to match. At inference, rescale by `1/s_a` then normalize to −23 LUFS (EBU R128) for perceptually consistent playback.

Ablation: removing RMS normalization at `s_a = 1` drops FD_PaSST 65.8 → 81.3 (drastic). At `s_a = 3` the gap closes (63.05 → 64.2 without RMS) but the combination remains best. The interpretation: amplitude lifting is a **mandatory preprocessing pair** for raw-waveform flow matching, and the paper explicitly attributes the loss-magnitude / Gaussian-prior mismatch as the failure mode.

## Architecture (MMDiT)

Standard MultiModal Diffusion Transformer (Esser et al. 2024; MMAudio backbone). Three streams enter the joint attention sequence: audio waveform tokens, frozen CLIP visual features, frozen CLIP text features. Audio and visual project to shared hidden dim via convolutional input blocks; text uses a linear projection.

- `L_joint = 4` (M) / `7` (L) joint multimodal blocks → `L_fused = 8` (M) / `14` (L) audio-only blocks → AdaLN + 1-D Conv (kernel 7) output → unpatchify to waveform.
- Hidden dim `d = 896`, 14 attention heads, both M (624 M params) and L (1.03 B params).
- **Dual-level conditioning.** Global `c_g` = sum of mean-pooled visual + text + timestep embedding (semantic guidance, *what*). Frame-aligned `c_c ∈ ℝ^{C×d}` = Synchformer per-segment features + per-segment positional embeddings, upsampled by nearest-neighbour to length `C`, then `c_g` added (temporal grounding, *when*). Injected via AdaLN-style modulation.
- **RoPE on Q/K only** (no text, since captions are unordered). The audio and visual streams run at different frame rates, so apply RoPE base-frequency ratio `C / N_clip` to visual queries (e.g. 10× for C = 640, N_clip = 64) — same temporal moment ↔ matching rotary phase across streams. This is the audio-domain version of the "multi-rate RoPE alignment" problem.

**Output head.** AdaLN + 1-D Conv (kernel 7) projects from `d` back to `D` samples per token. Note the *single* 1-D conv at the very end is reminiscent of LinDiff's Post-Conv smoother (3 Conv1d + 1 TA-LVC, 32 ch) but WavFlow's is *much* thinner — just one kernel-7 layer. Whether the patch-boundary harmonic artefacts LinDiff identified are suppressed by this thin head, or absorbed into the very high token count (C = 640–1764, vs LinDiff's max 3600 at 22 kHz), is not ablated.

## Training and inference

- Schedule: logit-normal `t` sampling (Esser et al. 2024). EMA decay 0.9999. AdamW (β₂ = 0.95), constant LR 1e-4, 20-epoch warmup, BF16. Gradient clip 1.0.
- 16 kHz models trained from scratch 400 epochs on 5 M / 1 M / 200 K mixtures, batch 10 752 (VT2A-L), 8 192 (T2A), 1 536 (200 K VGG only).
- 44.1 kHz Large is **SFT from converged 16 kHz Large**, LR 1e-5, 650 epochs, batch 1 536, on 200 K augmented VGGSound at 44.1 kHz. Same patchify D = 200 (so C = 1 764 instead of 640).
- Classifier-free guidance: 10% drop on both visual and text conditioning *independently and jointly*, with a learned null embedding. This is what lets the same checkpoint serve both V2A (full conditioning) and T2A (zero out visual via learned null).
- Inference: Euler ODE solver, 50 steps, CFG `ŵ = (1+w)·v(x,t,c) − w·v(x,t,∅)` with `w = 4.5`. Output rescaled by `1/s_a` and LUFS-normalised.

## Data scale matters more than usual

The paper's strongest empirical claim outside the architecture is that **raw-waveform training is exceptionally data-hungry compared to latent training** — there's no pretrained encoder to absorb the prior over acoustic structure, so the generator must learn it from scratch from paired data.

- 200 K (VGGSound only): FD_PaSST = 121.09 at D = 200 (Tab. 7); 96.91 at D = 256 in the patchify sweep (Tab. 3); both extremely poor.
- 1 M: drops to ~63.
- 3 M: ~60.
- 5 M (full): ~63 (saturation around 3–5 M).

Patchify granularity × data scale interact: at 200 K data, finer patches (D = 200) strictly help (`136 → 90` FD_PaSST). At 3 M data, the same fine patch saturates and **a coarser patch (D = 512) hits a capacity bottleneck** (degrades to 89 from 81 at 200 K — the only non-monotonic point in the sweep). The sweet spot at D = 200, C = 640 holds across scales because per-token information density is sufficient without exhausting transformer capacity. Default everywhere: D = 200.

**Project translation:** the project does not have 1 M paired waveform-text or waveform-video data — the training budget is closer to CRASH's ~30 k drum clips. WavFlow's data sensitivity at coarse patches suggests **the patch dim has to be smaller in low-data regimes** (closer to LinDiff's D = 64 than WavFlow's D = 200). The empirical question for the project is whether D = 64 at small data still avoids the capacity bottleneck WavFlow hits at D = 512 / 3 M.

## Curation pipeline (briefly)

Three-stage filter on a 50 M proprietary media corpus + open VGGSound (~100 K filtered) + open AudioCaps + Freesound (~150 K filtered):

1. **Filter** — 8 s windows, drop > 80% silence, drop bottom 10% PANN class confidence, drop audiobox-aesthetics PQ < 6.0.
2. **Balance** — align proprietary 50 M filtered down to 5 M by VGGSound class distribution.
3. **Augment** — small datasets get ×2 temporal augmentation (overlapping windows starting at 0 s and 1 s).

Two final mixtures: VT2A (5 M media + 200 K VGGSound), T2A (1 M media + 300 K open T2A). VGGSound captions are kept as native **sparse labels** (e.g. "dog barking") in the final mixture — mixing sparse + dense captions caused training divergence; rephrasing VGGSound to dense via Qwen3-Omni stabilised training but degraded DeSync/IB; sparse + visual modality as the grounding bridge was the working recipe.

## Ablations worth keeping

### Noise-level shift fails on raw audio (Appendix E)

Image-side practice biases training toward higher noise levels via `t_s = t / (t + s · (1 − t))` (Hoogeboom et al. 2023; Li & He 2025) — proven to help pixel-space transformer diffusion. WavFlow runs the same knob on waveform:

| s | FD_PaSST | FD_PANN | IS | DeSync |
|---|---|---|---|---|
| 1.0 (default) | **63.05** | **6.21** | **15.58** | **0.50** |
| 3.0 | 73.17 | 7.52 | 13.64 | 0.50 |
| 5.0 | 92.21 | 10.25 | 11.81 | 0.55 |

**All metrics degrade monotonically.** The paper's mechanistic explanation: image pixels span a broad dynamic range where high-noise training helps capture low-frequency global structure; audio waveforms have inherently low signal density even after the amplitude lift, so further noise burying makes the signal unrecoverable. The audio-side prescription: **don't carry over image-domain noise-shift practice**. Default `s = 1.0`.

This is a direct empirical contradiction of an image-domain best practice when ported to raw audio.

### Inference hyperparameter sweep

- CFG: 1.0 (none) → 9.86 IS, 0.71 DeSync (terrible). 2.5 → best FD_PaSST. 4.5 → best on most metrics, default. 7.0 → over-saturated, FD degrades.
- ODE steps: 10 → already most of the quality (10.73 FD_PANN). 25 → marginal gain. 50 → plateaus. 100 → no further gain. **The 50-NFE choice is conservative — 25 NFE is within 1 FD_PANN of 50.**

50 → 25 NFE roughly halves inference cost at competitive quality. A more aggressive 10-NFE deployment is plausibly within reach with mild post-training (CM / GAN-finetune on the FM teacher).

### Patch sync-alignment

Configurations specifically chosen to align audio token count with Synchformer feature length (192 tokens × integer multiples, e.g. C = 576 or 768) **did not help** beyond what generic fine-granularity gave. Robustness suggests positional alignment between conditioning and audio token streams is handled adequately by the RoPE base-frequency ratio + interpolated frame-aligned condition.

## Results headlines (for context, not focus)

VGGSound VT2A (FD_PaSST ↓): WavFlow-L-44.1kHz **55.82** beats MMAudio-L-44.1kHz 60.60; WavFlow-L-16kHz 59.98 essentially ties; DeSync ties (both 0.44–0.46). AudioCaps T2A: WavFlow-M-16kHz FD_PANN **10.63** (best), IS 12.62 (best), versus MMAudio-L-44.1kHz 15.04.

The takeaway is not the leaderboard win but the **equivalence-at-larger-scale** claim: raw-waveform FM with simple preprocessing matches latent SOTA on the same multimodal benchmark.

## CPU and project relevance

**No CPU benchmark in the paper.** Even with one, the architecture is unsuited to the project's CPU + 1-NFE target:

- 1.03 B params (or 624 M for M) — 5–10× over the largest existing audio CPU benchmark (Flow2GAN, 78.9 M, 4.85× RT at 1-NFE).
- Sequence length 640 tokens at 16 kHz (1 764 at 44.1 kHz) — large but not catastrophic per-step; the param count dominates.
- 50 NFE — multiplies any per-step number by 50.

Useful as a **recipe validation**, not a deployment artifact. The portable pieces:

1. **`x`-pred + `v`-loss combo on raw waveform** — confirmed audio-side. The pixel-MF recipe (which JiT/pMF prove for image) generalises to 1-D audio at scale. The project's `fm_baseline.yaml` should default to this combination.
2. **Amplitude lifting** (`r_* = 0.33`, `s_a = 3.0`, mono) — drop-in preprocessing for any raw-waveform FM, ~15 FD_PaSST swing isolated. Worth running at the project's scale to validate the magnitude of the gain on shorter clips and smaller models.
3. **Waveform patchify with D in the 64–256 range** — confirms 1-D patchify as the entry/exit machinery. At the project's smaller scale, expect to need smaller D than WavFlow's 200 to avoid the capacity bottleneck.
4. **Don't carry over noise-shift `t_s = t/(t + s(1−t))` from image practice.** Default `s = 1.0` (i.e. no shift) on raw audio.
5. **Thin output head (kernel-7 1-D Conv + AdaLN) suffices at this scale** — the question is whether it suffices at the project's much smaller scale where LinDiff needed a 3-Conv1d + TA-LVC smoother.

## Limitations and gaps

- **No CPU number, no GFLOPs, no inference latency table.** Cannot be situated against the project's CPU benchmark grid; only the architectural shape is portable.
- **No speech / singing.** Generated vocalisations are non-linguistic — the model never learned phoneme structure. Linguistic content needs finer patches and dedicated speech corpora.
- **No 1-NFE or few-NFE variant.** All results at 50 NFE; no post-training (CM, GAN, APT) for a few-step student. Untested whether the same backbone distils cleanly down to 1–4 NFE.
- **No ablation on the thin output head.** Whether the kernel-7 Conv1d is load-bearing for patch-boundary smoothing (as LinDiff's Post-Conv is) or vestigial is unclear.
- **Conditioning is mandatory.** Pure unconditional raw-waveform generation is not tested; CFG drop rate to zero means the model is trained to handle null conditions, but unconditional FAD is not reported.

## What this validates / contradicts in the wiki

- **Validates** the JiT / pMF pixel-space `x`-prediction recipe on raw audio at scale (see [representation-space.md](../concepts/representation-space.md), [methods.md](../concepts/methods.md) §FM design rules).
- **Adds** amplitude lifting as a third audio-specific FM design rule alongside `x_1`-prediction and spectral-energy-inverse loss weighting.
- **Contradicts** noise-shift recommendations carried over from image-domain pixel-space transformer diffusion. Worth flagging in any noise-schedule analysis for audio.
- **Validates** patched-ViT on raw audio as a scalable family (LinDiff was at vocoder MOS scale; WavFlow at multimodal-FD-leaderboard scale).
- **Extends** the [architecture.md](../concepts/architecture.md) patched-ViT-on-audio data point from "LinDiff is the only published instance" to "LinDiff + WavFlow span 4-layer / 256-d / vocoder to 30+ layer / 896-d / multimodal".
- **Open question now better-framed:** the project's "Variable-length via patch scaling" question — WavFlow does *not* test this (same `D` across different SR gives different token counts). Still genuinely open.

## Related work the paper anchors

- **MMAudio** (Cheng et al. 2025) — direct latent-space predecessor, same MMDiT backbone; WavFlow's baseline target.
- **HunyuanVideo-Foley** (Shan et al. 2025) — scaled-up latent V2A; WavFlow beats it at smaller param count.
- **JiT** (Li & He 2025) — image-domain `x`-prediction recipe WavFlow ports to audio.
- **Frieren** (Wang et al. 2024b), **V2A-Mapper** — earlier V2A baselines on VGGSound.
- **CRASH, DiffWave, WaveGrad, WaveNet** — early raw-waveform diffusion / AR vocoders; WavFlow positions as "this approach was abandoned for latent compression; we're going back to it with the right ingredients".
