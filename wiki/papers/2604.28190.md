# TL;DR — why this matters for raw-audio CPU gen

Fréchet Distance can be **optimised as a training loss**, not just used as an eval metric, by decoupling the population (~50k) used for the FD estimate from the batch (~1024) that carries gradients. Practical consequence directly relevant here: a **single-step** pixel/latent generator post-trained with FD-loss matches or beats multi-step baselines on ImageNet (0.72 FID @ 1 NFE), and **multi-step models can be repurposed to one-step** without any teacher distillation, adversarial training, or per-sample target. For our project, that is the most attractive distillation-free one-step pathway we have seen so far: it depends only on (a) a frozen feature extractor and (b) precomputed real-data feature moments, both of which have audio analogues (CLAP/VGGish/EnCodec features and FAD-style reference stats). It also reframes FAD: same gradient surgery would let us train against Fréchet-Audio-Distance-like objectives directly, not only score against them.

# What problem does it solve

Two coupled problems:

1. **FD as a loss has always been considered impractical** because reliable estimation of (μ, Σ) for d-dimensional features (e.g. 2048-d Inception) demands populations far larger than any training batch, and back-propping through a 50k-sample population is infeasible. Prior attempts (Mathiasen & Hvilshøj 2020; Doan et al. 2020) used small-batch FD and degraded base generators.
2. **FID has saturated as an evaluator.** SOTA ImageNet generators now beat the *real* validation set under Inception FID (val FID = 1.68; many models < 1.5) yet are still clearly distinguishable from real images. Inception is dated; modern self-supervised / VL backbones rank quality differently.

# Method

## FD-loss: the decoupling trick

The generator emits a small batch B (~1024). Features φ(x̂) for these B samples are *gathered* into a much larger statistics pool of size N (~50k–100k). FD is computed over the full pool against precomputed real moments (μ_r, Σ_r). **Gradients flow only through the current B features**; the rest of the pool is detached. Two implementations:

- **Queue estimator** (MoCo-style): FIFO of recent generated features of length N. Each step: enqueue new B features, dequeue oldest B. FD computed from queue empirical (μ_g, Σ_g). Stale features are treated as constants.
- **EMA estimator**: maintain running EMA of first and second feature moments with decay β. Recover Σ from the second moment minus outer product of mean. No queue storage, scales better when combining multiple representation backbones. Best β = 0.999.

The base loss is

  L = FD_φ(R, G) = ‖μ_r − μ_g‖² + Tr(Σ_r + Σ_g − 2 (Σ_r Σ_g)^{1/2})

Multi-representation variant — combine K backbones φ_i, normalising each term so it's unit-scale regardless of feature magnitude:

  L_φ_i = FD_φ_i(R, G) / sg(FD_φ_i(R, G) + c),   L = Σ w_i L_φ_i

with stop-gradient on the denominator. Equal weights w_i = 1 in the paper.

## FDr^k — a new evaluation metric

Because FD scales differ wildly across backbones, the *normalised* FD ratio for backbone φ_i is

  FDr_φ_i(G) = FD_φ_i(G, T) / FD_φ_i(V, T)

where T is the training set and V the validation set. By construction val images score exactly 1.0. **FDr^k** averages this ratio over K backbones; the paper uses K=6: Inception-v3, ConvNeXt-v2, DINOv2, MAE, SigLIP2, CLIP.

Crucial finding: under FDr^6, even SOTA generators score 3–6 (i.e. 3–6× farther from training than the val set is) — a much wider, less saturated quality gap than the FID-only view suggests.

# Architecture and inference cost

The paper is method-agnostic about the generator. It post-trains existing architectures:

- **pMF (pixel-MeanFlow)** — pixel-space one-step generator
- **iMF (improved MeanFlow)** — latent-space one-step generator
- **JiT** — multi-step pixel-space DiT-style, repurposed to 1-NFE
- **SD3.5 Medium** (2.5B MMDiT) — text-to-image, repurposed from 56 NFE to 1 NFE

Inference after post-training is **1 NFE** (single network evaluation). The training cost adds only the feature-extractor forward pass per sample and a small linear-algebra block (trace + matrix square root on a d×d covariance, d ≤ 2048). Backprop is through B samples only, so memory cost is essentially identical to ordinary training; the queue/EMA is a constant overhead.

# Headline results

- pMF-H + FD-loss (1 NFE, pixel space, 935M params): **FID 0.77, FDr^6 1.89** on ImageNet 256×256 — beats all prior multi- and one-step entries in the table.
- JiT-L base model: 1-step naive FID = 291.59 → after FD-loss post-training, **FID 0.85 / FDr^6 3.29** at 1 NFE. This is the distillation-free multi→one-step conversion.
- SD3.5 Medium 56 NFE → 1 NFE while preserving prompt content and inheriting reference-set stylisation. Demonstrated qualitatively only.
- Human preference: FD-loss post-trained pMF-H preferred 75.7% of the time against its base model; preferred 37.4% of the time against real validation images (so still not "solved").

Ablations worth remembering for audio transfer:

- Queue size sweep (Tab 1a): N = 50k optimal; N = 0 (batch-only) collapses to small-batch FD and degrades base. At N = 500k, FID still improves but FDr^6 *degrades* — early sign that FID alone misleads.
- EMA β sweep (Tab 1b): robust across β ∈ [0.9, 0.9999]; β = 0.999 best. EMA matches/beats queue with much less memory.
- Choice of feature space matters more than choice of estimator: optimising Inception alone gives best Inception-FID but worse perceptual quality than e.g. ConvNeXt or SigLIP2. Multi-rep (FD-SIM = SigLIP2+Inception+MAE) is the default.

# How this translates to raw audio

This is the most directly transferable image-domain method we've seen for our pipeline. The translation is almost line-for-line:

- **Frozen φ(x̂) for audio.** Direct candidates: VGGish (the original FAD backbone), CLAP, MERT, Wav2Vec2, EnCodec encoder embeddings, AudioMAE. The paper's own conclusion — that *which* representation matters more than the optimiser details — suggests we should multi-rep from the start. Equivalent of FD-SIM for audio: CLAP + AudioMAE + VGGish, perhaps.
- **Sample length implication.** φ for audio usually consumes 1–10 s clips and outputs a fixed-d vector (after temporal pooling). FD-loss only needs a fixed-d vector per sample → orthogonal to the variable clip lengths we care about (0.1–10 s). For very short clips (0.1 s) some backbones won't produce stable features; we should test which encoders accept ≥ 200 ms windows.
- **Reference statistics are cheap.** μ_r, Σ_r over a 50k clip corpus computed once, offline, on GPU. No real audio touches training — fits our pipeline (we train on 4090, infer on CPU; reference stats only impact training compute).
- **One-NFE inference is exactly our target.** Distillation-free multi-step → one-step conversion is the killer feature: we can train a small diffusion / flow-matching audio model with whatever objective converges fastest, then convert it to a 1-NFE generator for CPU inference. No teacher KD, no GAN instability.
- **CPU benchmark implication.** Inference cost is unchanged by FD-loss — same network, 1 NFE. The CPU latency of the *base* one-step generator is what we measure. Training cost adds feature extractor + d×d matrix ops; that's GPU-side and irrelevant to deployment.
- **Phase / waveform-level fidelity.** FD-loss is a *distributional* match in φ-space. If φ is invariant to phase (most audio classification embeddings are), the loss will not penalise phase artefacts. For raw sample-space gen, we likely need at least one φ trained on raw waveform reconstruction (EnCodec encoder, Wav2Vec2) in the multi-rep mix — analogous to MAE in the image FD-SIM combo.
- **Connection to FAD.** Fréchet Audio Distance = FD_VGGish. The paper makes FAD itself trainable. FAD has the same Inception-era staleness problem as FID; the paper's FDr^k story strongly suggests we should report FDr^k_audio (VGGish + CLAP + AudioMAE + ...) rather than headline FAD.

# Strengths

- **Distillation-free, GAN-free path to one-step**. No teacher model, no discriminator, no per-sample regression target.
- Works at scale (2.5B MMDiT post-trained successfully).
- Method-agnostic across pixel/latent/discrete spaces.
- Reframes FID-saturation as an *evaluator* problem, not "image gen is solved."
- Single hyper-knob (β or N) and otherwise drop-in.

# Limitations / things to test

- Gaussian moment-matching assumption. FD models φ(x) as Gaussian; with deep features that's a strong assumption and the paper acknowledges it.
- Multi-rep weighting is fixed equal — possibly suboptimal across audio backbones with very different signal-to-noise.
- Queue staleness window N: 50k images is one thing; 50k audio feature vectors is the same memory footprint but the staleness time depends on training batch throughput.
- No public audio replication yet (paper is image-only).
- For raw-sample audio at high SR, what does φ even look like for short (0.1 s) samples? Open.

# Open questions for this project

1. Which audio φ stack gives the best one-step CPU generator? Concrete experiment: train pMF-style 1D conv one-step generator at 22.05 kHz, 1 s clips; post-train with FD-loss using {VGGish, AudioMAE, CLAP} multi-rep; benchmark CPU latency and FDr^k.
2. Can we use a *small, in-house* φ trained on our own corpus instead of pretrained encoders? FD-loss only needs the moments to be well-defined; nothing requires φ to be a frontier model.
3. EMA vs queue scaling at audio sample sizes (1 s clips → cheaper features, can we push N higher)?
4. Multi→one-step conversion on a flow-matching audio model — does the same "naive 1-step → post-train" recipe work?
