## TL;DR (project-relevant)

**BAR is a discrete-tokenizer autoregressive image generator that bridges the historical "discrete loses to continuous" gap by scaling the FSQ codebook to extreme sizes (up to 2³²) and replacing the cross-entropy classification head with a small *masked bit modeling* (MBM) head that predicts the token's binary code progressively over 2–5 iterative unmasking steps.** **This is not a raw-pixel method and not a one-step method — it operates in a learned discrete latent (SigLIP2-initialised FSQ at 16× downsample of 256² images), and inference is a two-level loop: an outer 256-step autoregressive transformer pass + an inner ~2–5 MBM-head pass per token.** For this project the headline negative result is: the paper proves *discrete + scaled-bit-budget can match continuous in reconstruction and beat it in generation* — but it does so by introducing a learned tokenizer (the very codec floor this project rejects) and an `O(L · N)` inference loop, where `L = 256` tokens and `N = 2–5` MBM steps per token. **The portable trick is decoupling vocabulary size from prediction-head cost**: instead of a `O(C)` softmax over codebook size `C = 2ᵏ`, predict the `k` bits via a `O(k)` iterative-unmasking head. **Audio analogue would require both: (i) a discrete audio tokenizer with scaled bit budget (EnCodec/DAC at extreme codebook size, or a per-sample bit-plane decomposition of int16 PCM), and (ii) acceptance of two-level inference cost** — fundamentally incompatible with the project's "1-NFE on CPU faster than RT" target unless `L · N` collapses to something `O(1)`. **Useful as a contrastive data point for the wiki's representation-space framing**, not as a recipe.

## What BAR actually is

The paper's framing has two stages — a unification claim about discrete vs continuous tokenizers, and a generator recipe (BAR).

### The unification claim — Bit Budget unifies the two paradigms

Continuous and discrete tokenizers both reduce an image `I ∈ ℝ^{H×W×3}` to a latent `X` via `Encoder → Bottleneck → Decoder`. The only difference is the bottleneck:
- **Discrete:** vector-quantize `L = Encoder(I)` against a codebook of size `C`; latent `X` is a grid of indices, each carrying `⌈log₂ C⌉` bits.
- **Continuous:** dimensionality-reduction + KL-reg; latent `X` is a real-valued tensor with `D` channels, each carrying ~16 bits at mixed precision.

The paper defines a unified **Bit Budget** (Eq. 3–4):
```
B_discrete    = (H/f) × (W/f) × ⌈log₂ C⌉
B_continuous  = (H/f) × (W/f) × D × 16
```

At matched bit budget (e.g. MAR-VAE `D=16, f=16 ⇒ B=65536` vs LlamaGen-VQ `C=16384, f=16 ⇒ B=3584`), continuous wins reconstruction FID 0.53 vs 2.19 — but **the gap collapses once the discrete codebook is scaled up**. **Fig. 4: BAR-FSQ matches SD-VAE rFID at ~32K bits and surpasses MAR-VAE at ~64K bits.** **Discrete is not intrinsically inferior; it's been bit-budget-starved.**

This framing is what makes the paper's title claim plausible — they're not removing the tokenizer, they're scaling its bits.

### The generator recipe — BAR

Two functional components (Fig. 5):

1. **Autoregressive Transformer `F`** (the RAR backbone — Yu et al. 2025a): causal-attention transformer over the L=256 token sequence. Given prefix `{x_1, ..., x_{i-1}}`, outputs a per-position context vector `z_{i-1}`. This is unchanged from prior AR generators; standard transformer cost.
2. **Masked Bit Modeling head `G_θ` (MBM, the paper's novel piece)**: a small 3-layer SwiGLU+adaLN MLP that takes the context `z_{i-1}` and an initial all-masked bit-vector `Mask_bit(x_i)` (k mask tokens) and runs **iterative bitwise unmasking**:

```
Training (Eq. 6–7):
    Mask a random subset of bits in the ground-truth token's k-bit code with [MASK];
    L = (1/n) Σᵢ CrossEntropy_bit(x_i, MBM(Mask_bit(x_i) | z_{i-1}, M))

Inference:
    Initialise all k bits as [MASK];
    For N steps (N = 2..5, paper default 4):
        forward MBM once;
        unmask ⌈k/N⌉ bits with highest predicted confidence;
    Final 0/1 vector → token index → next-position embedding for the AR transformer.
```

This decouples vocabulary size from prediction head cost. The standard alternatives — both ablated and beaten:
- **Linear head**: monolithic softmax over `2^k` classes. Memory `O(2^k)`. Trains at codebook ≤ 2¹⁸, OOMs beyond. Quality good when it fits.
- **Bit head (Infinity-style)**: per-bit independent sigmoid, single forward pass. Memory `O(k)`. Scales to 2⁶⁴, but **quality degrades sharply at large codebook** (Fig. 6: rFID 0.5 at 64 bits but gFID > 10 without CFG; with CFG gFID 2.6 at 18 bits, worse at scale).
- **MBM (this paper)**: per-bit prediction *plus* iterative refinement via masking. Memory `O(k)`. Quality competitive with linear head at small codebook *and* scales monotonically beyond it.

### Sampling cost

Two nested loops:

```
for i in 1..L:                          # outer: AR transformer, L=256 calls
    z_{i-1} = F(x_1, ..., x_{i-1})      #   one full transformer forward
    bits = MASK                          #   inner: MBM head, N=2..5 calls
    for step in 1..N:
        bits = MBM(z_{i-1}, bits)        #     one tiny MLP forward
        unmask top-k/N predicted bits
    x_i = bits_to_index(bits)
```

Total NFE = `L × (1 transformer-forward + N MLP-forwards) = 256 × (1 + N)`. The MBM head is tiny (3-layer MLP at width 1024–2048) — the inner loop adds only marginal cost vs a single-forward linear head. The transformer dominates wall-clock. **Effective NFE for purposes of comparison with diffusion/FM: somewhere between 256 and 256·N depending on per-call cost, but the dominant axis is the AR loop length L.** KV-caching is used (mentioned in Tab. 4 caption); each outer step amortises but the sequence is still serial.

The paper compares this against:
- 1-step diffusion (MeanFlow): a single full-network forward, FID 2.20 / 676 M params / 151.48 img/s on H200.
- BAR-B (415 M params), 256 outer × ~5 inner ≈ 1280 effective MBM forwards + 256 transformer forwards: **24.33 img/s**.
- BAR-B/2 (token-shuffling 2× downsample, 64 outer steps, 16 bits/token): 150.52 img/s — matches MeanFlow throughput at higher quality (gFID 1.35 vs 2.20).
- BAR-B/4 (16 outer steps, 256 bits/token): 445.5 img/s — fastest variant, quality drops to gFID 2.34.

The **token-shuffling trade** (§4.2, Tab. 2d) lets BAR trade `L · k_per_token = const` — fewer outer steps, more bits per token. This is the paper's only lever for collapsing the outer AR loop.

## Where BAR sits in the literature this project tracks

### Vs MaskGIT / MAR / MaskBit / Infinity (the immediate neighbours)

| Method | Tokenizer | Generation paradigm | Per-token prediction | Outer-loop steps |
|---|---|---|---|---|
| MaskGIT | small VQ (~1k–10k) | iterative masked-token | softmax over `C` classes | 8–10 |
| MAR | continuous (no quantizer) | random-order AR + diffusion head | diffusion `O(64)` per token | 64 (MAR steps) |
| MaskBit | LFQ bit-tokens (~2¹⁰) | masked-token (MaskGIT-style) | feeds bits to AR transformer directly | ~10 |
| Infinity | BSQ (2⁶⁴) | VAR scale-by-scale + bit head + bit-corrector | per-bit sigmoid (`k=64` independent) + post-corrector | VAR-scale steps + corrector |
| **BAR** | **FSQ at extreme codebook (10–64 bits)** | **standard left-to-right AR + MBM head** | **iterative bitwise unmasking, 2–5 MBM forwards** | **L=256 (or 64 / 16 with shuffling)** |

**BAR's two contributions over this lineage:**
1. **Compatibility with vanilla left-to-right AR** (unlike MaskBit which uses MaskGIT-style ordering, or Infinity which is locked to VAR's scale-by-scale + corrector). The MBM head plugs onto any AR transformer.
2. **Iterative unmasking *within* a single token's bit-code** — bridges MaskGIT (iterative over tokens) and Infinity (parallel over bits within a token) by being iterative over bits within a token, conditioned on a globally-AR context.

### Vs continuous one-step / few-step (MeanFlow, pMF, JiT, FM)

| Axis | MeanFlow / iMF / pMF / FM | **BAR** |
|---|---|---|
| Operating space | continuous (VAE-latent or pixel) | **discrete (FSQ-tokenized, can be very large codebook)** |
| Inference shape | 1–4 full network forwards | **256 transformer forwards + ~1280 tiny MLP forwards** |
| Latent dim | continuous 16-D × 16×16 = 4096 | discrete 256 tokens × `k` bits |
| Quality at compute-matched scale (ImageNet-256) | iMF-XL/2 1.72, pMF-H/16 2.22, MeanFlow 2.20 | BAR-B 1.13, BAR-L 0.99 |
| Tokenizer-free? | **Yes for pMF (pixel), no for iMF/MeanFlow (VAE)** | No — needs FSQ tokenizer (SigLIP2-init, GAN-trained) |
| Compatible with this project's CPU target? | pMF: yes (pixel + 1-NFE + perceptual loss). iMF/MF: latent-only. | **No — `256 × N` serial outer loop, plus a frozen tokenizer + frozen detokenizer in the inference path** |

**The critical comparison is BAR vs pMF.** pMF runs **one** 271-GFLOP forward in raw pixel space (no tokenizer at training or inference) at FID 2.22. BAR runs 256 transformer forwards plus a tokenizer encode + decode at FID 0.99–1.13. The compute cost of BAR is much higher; the quality is also much higher. **The Pareto trade for short audio CPU is essentially decided** — pMF is the project's target shape; BAR's recipe is interesting but on the wrong side of the cost-quality frontier *for this project's constraints*.

### Vs raw-pixel diffusion (ADM, JiT, SiD2)

ADM at 554 M, 1000 NFE, FID 3.94 with CFG. JiT at 2 B, FID 1.82. **BAR is much cheaper than ADM and matches JiT at one-fourth the params.** This is the cleanest existence proof in the recent literature that "discrete with enough bits" can match "continuous in raw pixels with enough compute" — the BAR paper's Fig. 4 reconstruction sweep is in particular the strongest evidence that a *learned* discrete tokenizer with `~2³²` codes can preserve quality fully.

## Is BAR tokenizer-free? Critical for this project — answer: **no**

The paper is explicit (§3.2, §4.1):
- BAR uses a **discrete tokenizer (BAR-FSQ)** at `f = 16` downsample of 256² images → 16×16 = **256 tokens per image**.
- Tokenizer encoder is **initialised from SigLIP2-so400M** (a foundation model trained with vision-language alignment objectives).
- Tokenizer is trained with a complex multi-loss recipe: L1 + L2 + perceptual + Gram + GAN, using a frozen DINO model as discriminator. 40 epochs ablation training, plus 40 epochs of decoder-only finetuning for final models.
- **The tokenizer is a hard prerequisite. The whole "bit budget" framing is a property of the tokenizer, not of the generator.**

This is the **2-stage codec floor** that the project's [representation-space.md](../wiki/concepts/representation-space.md) framework identifies as the central trade. BAR pushes the codec floor up (rFID 0.33 at 65536-bit budget — better than SD-VAE 0.62) but does not remove it.

### Could a hypothetical "tokenizer-free BAR" exist for audio?

Yes, conceptually. The audio analogue would be:
- **Bit-plane decomposition of raw int16 PCM**: each sample is 16 bits, naturally bit-decomposable. A 1 s @ 22 kHz mono clip = 22050 samples × 16 bits = 352800 bits ≈ 43 KB.
- Sequence length: 22050 outer steps if we treat each sample as a "token"; even with patch-shuffling at p=64 (analogous to BAR's `BAR-B/4`), still ~344 patches.
- Or analog-bits (Chen et al., NeurIPS 2022): treat the 16 bit-planes as a 16-channel "image" and run continuous diffusion in `[-1, +1]^{T×16}` — a *different* paradigm, currently filed under "bit-level continuous diffusion", not BAR-style discrete AR.

**Neither of these is what BAR does.** BAR's "bits" are bits of an **FSQ-codebook index** that has been *learned* to factor an image into a 256-token grid where each token carries `k` informative bits. The "bit modelling" is operating on a learned discrete latent, not on the raw bytes of the data. Calling BAR's representation "bits" is a slightly misleading name from a raw-data perspective.

## Inference cost characterisation (the project's headline lens)

### Throughput (paper Tab. 4, H200, float32, KV-cache, no further optimisation)

| Method | Params | NFE shape | img/s | FID (CFG) |
|---|---|---|---|---|
| PAR-4× | 3.1 B | parallelized AR | 4.92 | 2.29 |
| VAR | 2.0 B | scale-by-scale | 8.08 | 1.92 |
| MeanFlow | 676 M | 1 (one continuous forward) | 151.48 | 2.20 |
| **BAR-B/4** | 416 M | 16 outer × ~5 inner | **445.48** | 2.34 |
| **BAR-B/2** | 415 M | 64 outer × ~5 inner | **150.52** | 1.35 |
| BAR-B | 415 M | 256 outer × ~5 inner | 24.33 | 1.13 |
| BAR-L | 1.1 B | 256 outer × ~5 inner | 10.65 | 0.99 |

**The key cost shape**: BAR-B/4 at 16 outer steps is faster than MeanFlow's 1-step on H200 *for the same model size* because the per-step transformer is smaller than MeanFlow's full 676 M backbone *and* 16 KV-cached AR steps amortise well on GPU. The BAR-B/2 row is the wiki's relevant data point — **at 150 img/s (matched to MeanFlow throughput) BAR-B/2 beats MeanFlow on FID 1.35 vs 2.20**.

**But:** no paper-reported CPU number; AR loops generally do not amortise on CPU the way they do on GPU (latency-bound, not throughput-bound). The 1.0× RT target on CPU at audio sample rates means single-sample latency, not batched throughput.

### Why this is bad for CPU + audio

1. **Serial outer loop.** 256 transformer forwards in sequence — KV-cache makes each cheap on GPU (re-uses past states), but on CPU each forward incurs the full L1/L2 traffic of the model weights. **For an interactive CPU generator targeting 1 s @ 22 kHz with patch-shuffle to 256 tokens, that's 256 transformer-forward latencies in series.**
2. **No 1-NFE option at competitive quality.** The fastest BAR variant (B/4 @ 16 outer) trades quality (gFID 2.34) for speed and *still has 16 serial forwards*. There is no `L = 1` BAR — the paradigm fundamentally requires `L ≥ 16` for the bit budget structure to work.
3. **Tokenizer + detokenizer in the inference path.** Each generation requires a frozen encoder (training-time only) but a *decoder* at every inference call. The BAR decoder is a ViT-L architecture from scratch (~300 M params); that's a single forward per sample (after the AR loop completes). **Adds a fixed-cost floor independent of the AR loop length.**

For 0.1–10 s audio at 22 kHz mono, even the most aggressive BAR-style port — token-shuffle to 64 tokens, MBM head with 5 steps, small 50 M transformer — yields 64 transformer forwards + 64 × 5 = 320 MLP forwards + 1 detokenizer forward per generated clip. At any CPU latency the AR loop dominates. **Even optimistically, this is 10–100× slower than a 1-NFE pMF-style port.**

## How (and whether) this translates to raw audio

### What's audio-portable

1. **The Bit Budget framing.** Discrete audio tokenizers (EnCodec, DAC, SoundStream) all sit at modest codebook sizes (1024–16384 per codebook × 4–32 codebooks). The paper's central claim — *codebook scaling closes the reconstruction-quality gap to continuous tokenizers* — has not been tested for audio. **A scaled-FSQ audio tokenizer (e.g. `f = 320, C = 2³²` per token at 50 Hz token rate) is a plausible engineering target**, though no existing work does this. **The reconstruction-fidelity-vs-bit-budget sweep (BAR Fig. 4) is the most-portable single experiment** if anyone wanted to build a competitive audio-discrete tokenizer.
2. **The MBM head as a generic vocabulary-scaling primitive.** Independent of BAR's specific recipe, MBM is a drop-in replacement for any softmax-over-codebook prediction head when the vocabulary is large. Audio LMs (MusicGen, AudioLM, VALL-E) running on RVQ stacks typically have `4 × 1024 = 4096` per-position vocabulary; the MBM head's value would be small unless someone is also trying to scale codebook size. Composes with any audio AR LM that has a vocabulary-scaling problem.
3. **Token shuffling for trading sequence length vs per-token bit-count.** Direct audio analogue: at fixed total bit budget, fewer tokens × more bits per token. For audio at 50 Hz × 32 bits/token = 1600 bits/sec vs 12.5 Hz × 128 bits/token = same 1600 bits/sec — different inference cost shape. Worth trying if anyone is doing discrete-audio generation, but **the project's pMF-style path doesn't go through this regime**.

### What's not portable

1. **The whole AR-loop structure.** Audio at sample rate is ~10² – 10³ longer than 256 image tokens; at frame rate it's 50 Hz × 10 s = 500 tokens, which is comparable but still serial. Either way, the outer loop kills CPU latency.
2. **The two-stage tokenizer + generator pipeline.** This *is* the codec floor that the project rejects. Audio codecs (EnCodec at low bitrate, DAC) have audible reconstruction artefacts; scaling them to 2³² codes is non-trivial and would still inherit the music-vs-speech distribution issue most audio codecs have.
3. **The bit-modelling per se for raw audio.** Treating int16 audio samples as "16 bit-planes" and running a BAR-style head over the bit-planes is a *different* idea (analog-bits diffusion of Chen et al. 2022) and is not what BAR does — BAR's bits are bits of a learned discrete *index*, not bits of the data byte.

### How "bits" differ between BAR (images) and a hypothetical raw-audio bit method

| Aspect | BAR (images) | Hypothetical raw-audio bit method (e.g. analog-bits) |
|---|---|---|
| What the bits represent | Index into a learned FSQ codebook | Two's-complement bits of int16 sample value (or equivalent quantization of a float waveform) |
| Tokenizer required | Yes (learned, GAN-trained) | No |
| Number of bits per "token" | 10 to 64 (FSQ codebook size knob) | 16 (int16 PCM, fixed) |
| Position in sequence | 256 grid cells over a 256² image | 22050 samples per second of audio |
| Bit-plane semantics | None — bits are an arbitrary code of a learned codebook entry | **MSB carries envelope; LSB carries dither/noise** — a natural perceptual hierarchy |
| Iterative refinement makes sense? | Yes, but for codebook-bit prediction conditioned on context | Yes, MSB-to-LSB ordering has perceptual meaning |
| Existing precedent | BAR (this paper), MaskBit, Infinity | Chen et al. "Analog Bits" (NeurIPS 2022), Bit-level audio CNN (Wavenet-style 16-class softmax over μ-law) |

**The audio bit-plane idea is structurally different and structurally more interesting for the project's goal — but BAR does not address it.** If the project wanted to pursue a "bit-modelling" path for audio, the relevant prior art is **Chen et al. 2022 (Analog Bits)**, **WaveNet's 256-way μ-law softmax**, and **bit-level diffusion** — *not* BAR. BAR's name is misleading from an audio-perspective.

## Strengths and limitations

### Strengths

- Clean unification of the discrete-vs-continuous tokenizer debate via the Bit Budget metric. This is the paper's most-portable contribution.
- MBM head is a genuinely-novel and scalable solution to the "soft-max-over-2³² classes" problem; should be adopted in any AR generator with extreme-vocabulary needs.
- BAR-B/2 at 150 img/s, 415 M, FID 1.35 is a strong cost-quality point on ImageNet — beats MeanFlow at matched throughput.
- Token shuffling is a clean axis to trade quality for speed (BAR-B/4 at 445 img/s, FID 2.34 is the fastest competitive variant).

### Limitations for this project

- **Discrete-tokenizer-dependent.** The whole paradigm sits inside [representation-space.md](../wiki/concepts/representation-space.md)'s "latent / learned-tokenizer" cell, with the additional cost that the tokenizer also has a sophisticated GAN training stage (DINO-discriminator).
- **AR-loop inference shape.** Serial loop of 16–256 transformer forwards is incompatible with CPU latency budgets for short audio.
- **No CPU number reported.** Standard image-paper omission; throughput-only on H200.
- **No 1-NFE variant.** The paradigm requires `L ≥ 16`.

### Open questions (audio-side, untested)

- **Bit Budget sweep for audio tokenizers.** Does the codebook-scaling argument (Fig. 4) hold for audio tokenizers like EnCodec / DAC / SoundStream? Smallest experiment: train FSQ-quantized audio autoencoder at varying codebook sizes (2¹⁰ to 2³²); measure ViSQOL or MUSHRA reconstruction quality. If yes, the audio-discrete-vs-continuous debate gets a clean axis like the image side.
- **MBM head inside an existing audio LM.** Replace the per-codebook softmax in MusicGen / AudioLM with an MBM head; check whether quality holds at codebook-size sweep. Cheap drop-in test, scales to large codebooks for free.
- **Token shuffling in audio AR generation.** Existing audio AR generators run at codec frame rate (~50 Hz × 1024 vocab). Could we trade to ~6 Hz × 65536 vocab via token-shuffling? Inference shape changes but not in a way that obviously helps CPU.
- **Why does BAR's MBM converge so quickly?** Paper claims 400 epochs at batch 2048 for full BAR-L; 80 epochs for ablations. That's 3.68× faster than RAR at matched compute (paper's framing). Convergence rate matters for our 4090 budget — if the bit-iteration recipe genuinely helps optimization, the per-bit decomposition may have favourable loss-landscape properties.
